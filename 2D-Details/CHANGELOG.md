# Changelog

All notable changes to StructDraw from V25.modular onwards. Pre-V25 history (V0 → V24.A3) predates this changelog and was not migrated into the repo.

The format is loosely based on [Keep a Changelog](https://keepachangelog.com/), but pragmatic — entries are by phase / dev session, not by semver.

## 2026-06-12 — CHS grade + finish dropdowns driven by the Austube Mills availability guide

- **The CHS options bar (placement and selected-member) gains a Grade dropdown before Section** — C350L0 / C250L0, defaulting to C350L0 — **and a Finish dropdown after it** (Clear / Oiled / Hot-dip gal / DuraGalPlus). The Section list filters to the grade, each option labelled with the guide's nominal size + wall class (e.g. `88.9x3.2 · 80 L`), and the Finish list shows only the finishes actually offered for that exact size, with the supply route flagged in the label (ex-stock plain, `(ex-rolling)`, `(MOQ)`). Switching grade migrates the size to its same-OD equivalent in the other grade (50 M ↔ 50 L stays a DN50 pipe). Grade + finish are stamped onto the placed entity and editable in the inspector too. `js/72-v25-options-bar.js`, `js/69-v25-dispatch.js`, `js/71-v25-selection.js`.
- **CHS size lists now offer only what can actually be bought** — a new `CHS_AVAIL` catalogue (`js/02-data-sections.js`) transcribed from the Austube Mills Product Availability Guide (Apr 2018, pipe & tube structural products, pp. 3–5): 20 C350L0 sizes (Extra Light + Light pipe) and 24 C250L0 sizes (Medium / Heavy / Extra Heavy), 33.7–165.1 OD, each with its per-finish availability. The Pick… size picker groups by grade instead of OD; picking a size from the other grade's group drags the member's grade with it. The full 112-size DCT `CHS_DB` stays untouched for geometry, so existing saves (including >165.1 OD sizes, which Austube doesn't roll) still render and are prepended to the dropdown when selected. Default CHS size moved `88.9x5.9` → `88.9x3.2` (80 L, C350L0, clear ex-stock).
- Browser-verified: armed-tool bar order (Grade → Section → Finish), grade switch + same-OD migration, finish revalidation on size change (150 M flips Clear → Oiled, its only stocked finish), real two-click placement stamping `grade`/`finish`, selected-member bar + inspector mirrors, grade-grouped picker (20 + 24), clean console.

## 2026-06-12 — Fix: vector PDF export — hatches stay inside their shapes, title block prints

- **Hatch and stipple patterns no longer spill across the exported PDF.** The vector-PDF canvas shim treated `clip()` as a no-op (a V15-era shortcut from before the V25 material patterns existed), so a polygon hatch region exported as a sheet-wide 45° hatch band and a stippled bounding-box rectangle. The shim now emits real PDF clipping (`q` + path + `W n` at `clip()`, with the matching `Q` popped by the canvas `restore()` that ends the clip in on-screen rendering — including even-odd clips for hollow-section hatching). Everything that relies on clip exports correctly now: V25 mat stipple/hatch/concrete-aggregate, steel cross-hatch on irregular polygons, hollow-wall hatching, snapshot crops. `createPdfCanvasShim` in `js/44-pdf-export.js`; the multi-sheet exporter (`js/51`) shares the shim and is fixed by the same change.
- **The Bligh Tanner title block now appears in exported PDFs.** The baked title-block templates loaded their PNGs from file paths; when the app runs from `file://` the browser taints any canvas they're drawn into, so the export's `toDataURL` embed threw a (silently caught) SecurityError and the whole template vanished from the PDF — only the editable values printed. The three baked backgrounds are now embedded as data-URLs in a new generated file `js/87a-titleblock-bg-data.js` (regeneration instructions in its header), which never taints — fixing the PDF embed *and* the on-screen white→transparent keying under `file://`. `js/87-titleblock.js` prefers the embedded data and falls back to the file paths.
- **The export shim's `drawImage` now honours rotation, scale and source-crop.** Previously a rotated snapshot image exported unrotated and the 9-arg source-crop form was ignored; both now bake through a temp canvas with the full current transform. `js/44-pdf-export.js`.
- Browser-verified: shim unit test (balanced `q`/`Q`, `W` / `W*` clip ops, post-restore draws unclipped), full `exportSheetToPDF` run with a 45°-hatch triangle + concrete poly blob (pdf.js render of the captured output: hatch and stipple contained exactly to their polygons, zero stray ink, title block region ink-verified), clean console.

## 2026-06-11 — Multi-select drag (Bluebeam-style)

- **Dragging any member of a multi-selection now moves the whole selection together** — select several members/fixings (marquee or Shift-click), grab any one of them by the body, and they all slide by the same delta, no grouping needed (the Bluebeam behaviour). A plain click on an already-multi-selected entity keeps the selection instead of collapsing it; clicking an unselected entity still replaces the selection as before. Grip / rotate / vertex handles keep editing just their own entity. Plays nicely with everything around it: Shift mid-drag ortho-locks the whole set, Alt/Ctrl copy-drag duplicates and drags the copies (already multi-aware), a group inside the selection moves once (not double-translated via its group hook), the single-entity soft-snap is paused during a multi-drag (it would tear the relative layout apart), and the move records as **one atomic undo step** across every selected entity. New `v25SnapshotMoveTargetsMulti` + mousedown/move/up wiring in `js/39-events.js`. Browser-verified through real mouse events: 3-entity drag (equal deltas), atomic undo/redo, grouped-subset drag, single-drag and selection-collapse regressions, clean console.

## 2026-06-11 — Fix: stipple/concrete hatches no longer slow the whole canvas down

- **Sheets with a blockwork section wall (grout stipple) or concrete hatch scroll smoothly again.** Every wheel tick / mouse move redraws the whole sheet, and the stipple-style fills were issuing one canvas `fill()` per dot — up to 6,000 for a wall's grout stipple, plus 800 fill+stroke pairs and 2,400 dots for a concrete region — which dropped scrolling to single-digit fps on slower work machines. The dots / aggregate triangles / cross-hatch lines now accumulate into **one path per pattern** with a single fill/stroke: measured **4.0×** faster for the wall section strip, **2.2×** for concrete, **6.6×** for the grout/sand `mat` fills, with pixel-identical output (same seeds, same dots; overlapping translucent dots no longer double-darken, an imperceptible evening-out of the stipple tone). Vector PDF export keeps the per-dot path so the export shim still emits crisp circle primitives. `drawBlockWallSec2D` (`js/68`), `drawMat2D` concrete/dots branches (`js/67`). Browser-verified: old-vs-new benchmark on identical entities, pixel-fingerprint parity, clean console.

## 2026-06-11 — Free rotation for cross-sections & fixings

- **The inspector's Rotation° field now actually tilts a cross-section member.** Select (or double-click) any section-aspect member — SHS, RHS, UB, UC, PFC, WB, GLT — type e.g. **30** into **Rotation°** in the Settings tab, and the glyph spins 30° about its centre (an SHS at 45° reads as a diamond), centrelines and all. Previously sections only honoured the quarter-turn **Roll°** and silently ignored Rotation°. Roll and Rotation compose (roll picks the face, rotation tilts it), so inclined-wall details (raking sheeting, sloped purlins) can show every member at the true angle. Legacy saves that stored their spin in `rot` pre-Roll° are detected and not double-rotated. New `v25Mem2SecRotDeg` in `js/68-v25-tools.js`; renderer, world outline, selection halo (`js/71`), GLT notch frame (`js/72m`) and DXF (via the shared outline) all follow.
- **The to-scale fixings — bolt (bolt2), HBS/VGS screws, ChemSet stud — now support free rotation too.** Each already had a Rotation° field in the inspector; it now rigidly spins the drawn glyph about its anchor point (head/washer/thread and all), in 2D canvas, vector PDF (the shim's transform support) and DXF. Hit-testing, hover halo, selection highlight, marquee bounds and the stud's embedment tip/edge grips all track the rotated glyph (`v25FixingRotPt` / `v25FixingUnrotPt` in `js/66`, draw wrap in `js/69`, selection plumbing in `js/71`, DXF `place()` rotation in `js/45`). Bearing/clamp auto-detection still scans axis-aligned — on a rotated fixing it simply falls back to the placed point.

## 2026-06-08 — Fix: CLT panels couldn't be moved

- **CLT panels can now be body-dragged to reposition them.** Previously a placed CLT panel could be selected and grip-edited (rotate / resize / stretch) but not moved — clicking and dragging the panel body did nothing. The `clt` branch in `v25Move` (`js/71-v25-selection.js`) handled every grip case and then `return`ed unconditionally, so a `handle === 'body'` drag never reached the generic body-translate fallback. ## 2026-06-07 — Linework upgrade: the line (`l`) & polyline (`p`) tools

- **The basic line tools now feel like every other upgraded member.** In 2D mode **`l`** draws a **single straight segment** (click start, click end → committed and auto-selected, tool returns to Select) and **`p`** draws a **multi-vertex polyline** (click each node; Enter / right-click / click the first vertex to close). Both produce the same fully-editable `lineSet`, and a freshly drawn line is now **immediately selectable** — the previous problem (in 2D mode `l`/`p` fired the legacy 3D line tools, leaving an unselectable line) is fixed: those keys now route to the V25 line tool in 2D and only fall back to the legacy 3D tools on a 3D sheet.
- **Full node editing.** Click a line's stroke to **move the whole line** (Shift = H/V ortho lock); drag any **node** to move it (Shift = snap that segment to 0/45/90° off its neighbour); **Shift+click a segment** inserts a new node there (and lets you drag it straight away); **Shift+click a node** deletes it (kept ≥ 2 points). Every edit is a single undo step. **Double-click** a line opens the Settings tab (the old closed-line fill-picker shortcut is gone — fill now lives in the controls below).
- **A great editable control set, top-middle and on the left.** Selecting a line (or arming the tool) surfaces the same rich controls in the **top options bar** and the **left inspector**: **Width** (mm), **Style** (solid / dashed / dotted / centre / phantom), **Colour** (+ reset to theme), **Opacity**, one-click **AS 1100 named-type presets** (Visible / Hidden / Centre / Phantom — sets weight + style together), **arrowheads / end-caps** at each end (none / arrow / dot / AS 1100 tick), **line cap** (butt / round / square) and **join** (miter / round / bevel), and for a closed polyline a **Closed** toggle with **solid-colour or material-hatch fill**. The next-drawn line inherits whatever style is set in the bar.
- New canonical `lineSet` fields `cap` / `join` / `arrowStart` / `arrowEnd` (raw `lw` mm is the single width source; legacy `lwLevel` still honoured for old saves). Arrowheads/caps/joins render in `drawLineSet2D` (`js/70`, new `_drawLineSetEnds` helper — arrows never drawn on a closed path). Shared control builder/wiring `v25LineControlsHTML` / `v25LineControlsWire` + selected-line bar + placement bar (`js/72`); single-line mode + style-stamp at every commit site (`js/69`); 2D keyboard routing (`js/42`); style latches (`js/65`); node insert/delete + vertex 45° snap (`js/39`, `js/71`); inspector field set (`js/71`); DXF `lineSet` → LWPOLYLINE on layer `0` (`js/45`). Save/load automatic via `entities2D` JSON; old lines render unchanged. Browser-verified: l/p draw + auto-select, body/vertex/insert/delete/45°-snap through the real event handlers, options bar + inspector live two-way binding, AS 1100 presets, arrowhead/cap/join/fill rendering, DXF emit, save round-trip, poly-mat + 3D-tool regressions, clean console.

## 2026-06-07 — 2D members auto-order on overlap (first-drawn goes behind)

- **Overlapping 2D members now hidden-line themselves automatically.** When you draw a member over an existing one, the **newer (second-drawn) member sits in front by default**, so the older (first-drawn) member's covered span is rendered as an AS 1100 dashed hidden line with no manual step — the previous "select the behind member and Send to Back" chore is now the default. Depth was opt-in before (everything `z = 0`, nothing dashed until you used the right-click menu); now a newly placed member that **genuinely overlaps** another auto-bumps its `z` just above the highest member it covers. The right-click **Send to Back / Bring to Front** menu is unchanged, so flipping the order back is still one click on the second member.
- **Only true overlaps trigger it — joints stay solid.** Overlap is tested with the same strict-interior rule (0.5 mm flush tolerance) the dashing engine uses, so members that merely **abut at a joint** (shared flush face) do not dash, and members that don't touch are untouched (stay coplanar `z = 0`). A new member also auto-fronts over an existing member previously brought forward, and stacks correctly for 3+ overlapping members drawn in sequence.
- New `v25DepthAutoBackOnPlace` + `v25DepthPolysOverlap` (perimeter-sample overlap with a centroid-containment fallback for exact-duplicate/contained members) in `js/72h-v25-depth-order.js`; one hook in `v25Add` (`js/66-v25-helpers-frame.js`) run **before** `addEnt2D` so the chosen `z` is captured by the undo snapshot and round-trips through redo/save. Browser-verified by numeric capture through the real `v25Add` path: beam×column cross → 2nd member `z=1`, 1st member gains the 2nd as an occluder (dashes where covered) and stays at `z=0`; flush-abut and far-apart pairs stay unset; exact-duplicate stack → `z=2`; overlap onto a `z=7` member → `z=8`; small member fully inside another → in front; clean console.

## 2026-06-07 — GLT members: adjustable outer-edge line weight

- **GLT (timber) members now draw a heavier outer edge, with a slider to tune it.** Timber perimeters read heavier than steel, so the member outline is scaled by a new per-member **Edge** knob (0–100%), surfaced as a slider in the top options bar alongside Grade / Grain size / Spacing / Opacity (both at placement and when a GLT member is selected). 0% keeps the old steel weight (1.0×), 100% doubles it (2.0×); the **default is 50% (1.5×)** — a bit thicker than steel but never as heavy as the timber-hatch boundary. Existing GLT members with no saved value pick up the 1.5× default automatically. The multiplier scales whichever base weight applies — `VIS_HEAVY` in elevation, `CUT` in section — so section cuts stay heavier than elevation edges. New data/helper `GLT_EDGE_DEFAULT` / `gltEdgeMult` in `js/02h-data-glt.js`; render in `js/68-v25-tools.js`; slider build/wire/persist in `js/72k-v25-glt.js`; placement latch + stamp in `js/69-v25-dispatch.js`. Browser-verified by numeric draw-capture: elevation outline 0.5 → 0.75 (default) → 1.0 px as Edge goes 0 → 50 → 100, grain (0.15) and centreline (0.25) unchanged; section cut 1.0 → 2.0; slider live in the bar; clean console.

## 2026-06-07 — GLT members no longer auto-weld

- **Timber (GLT) members never get auto-weld hatching.** Timber is bolted/screwed, not welded, so a GLT member abutting steel — or another GLT member — no longer draws the 45° fillet-weld ticks at the contact line. `computeV25WeldInterfaces` now excludes `memberType === 'glt'` members from the weld-face collection, so timber contributes no weld faces in any combination. Steel-to-steel and plate-to-steel auto-welds are unaffected. One-line guard in `js/68-v25-tools.js`. Browser-verified in 2D mode: steel↔steel → 1 weld interface (unchanged), timber↔timber → 0, timber↔steel → 0; clean console.

## 2026-06-07 — Title blocks: re-baked from clean blank office templates

- **Timber‑screw connection feature — Phase 4: rule engine (`dev/` only) — v1 acceptance gate MET**. Algorithm walks the nine steps from `PlannedBuilds/timber-screws/03-rule-engine.md` exactly: (1) resolve rule context — picks the right `RuleSet` and `CapacityTable` from the catalogue based on `connectionType` × substrate family × `preDrilled`, looks up `k_mod` from EN 1995-1-1 Table 3.1 via service class × load duration, looks up `k_dens` from the rear member's `TimberClass`, fails fast with a clear `failingItems` entry if any lookup misses; (2) build geometric reference frame — normalises `rear.grainDir` to unit `grain`, derives `perp` as 90° CCW, normalises `connection.load.F_d_dir` to `Fhat`, computes `α` as `angleBetweenSymmetric(Fhat, grain)` normalised to [0, 90°] (so the engine is symmetric under load-direction reversal — the spacing rules don't care which way along grain the load points), records `F·grain` and `F·perp` for the inspector; (3) classify each of the four AABB edges via `classifyEdgeByLoad(Fhat, edgeNormal)` from `02e-catalogue-lookups.js` — strict dot-product test, boundary case (F·n exactly zero) classes as 'unloaded' to match Dan's hand calc and the Rothoblaas p. 215 figure ranges; (4) resolve required distances by linearly interpolating each of a₁/a₂/a₃,t/a₃,c/a₄,t/a₄,c between the rule set's α=0° and α=90° endpoint formulas via `interpRuleAtAlpha`, then applying Douglas-fir × 1.5 on a₁/a₃ and timber-to-timber × 1.5 on a₁/a₂ in the documented order; (5) per-screw edge-distance checks — for each screw × each edge, project the screw onto the AABB face along the outward normal to get the applied distance, look up the required value based on the edge's structural classification (end → a3t/a3c; side → a4t/a4c), emit a structured row with `applied`, `required`, `formula`, `pass`, `severity`; (6) pair-wise spacing — for every (i, j) pair project the inter-screw vector into the grain frame to get Δ_along and Δ_cross, check the EN 1995-1-1 / Rothoblaas rule that a pair PASSES if either |Δ_along| ≥ a₁_req OR |Δ_cross| ≥ a₂_req (not both — only fails if inside the (a₁, a₂) clearance rectangle in the grain frame), records the governing condition for the inspector; (7) n_ef row detection by clustering screws on their grain-projected `cross` coordinate with ε = a₂_req/2 (so two screws in the same row along grain land in the same cluster, two in different rows do not), sorting each cluster along `along`, finding the minimum consecutive spacing as a₁, looking up `lerpNef(n, a₁/d)` from the Rothoblaas table for n ∈ [2,5] with the EN 1995-1-1 §8.3.1.1 formula extension for n > 5;
