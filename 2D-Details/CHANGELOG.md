# Changelog

All notable changes to StructDraw from V25.modular onwards. Pre-V25 history (V0 → V24.A3) predates this changelog and was not migrated into the repo.

The format is loosely based on [Keep a Changelog](https://keepachangelog.com/), but pragmatic — entries are by phase / dev session, not by semver.

## 2026-06-03 — Rothoblaas HBS Plate timber screw: new 2D fixing (Draw tab → HBS)

A new timber-screw fixing for 2D paper-space mode, mirroring the bolt: pick **HBS** in the Draw-tab Members palette, choose a standard orientation (plan / section, each direction), and single-click to place. Drawn to true scale for every standard Rothoblaas HBS Plate length (Ø8 / Ø10 / Ø12, all 18 SKUs, from the verified ETA-11/0030 catalogue). Browser-verified 2026-06-03 (app boots console-clean; the section profile geometry was captured during a live draw and checked exact against the catalogue; the head-to-face snap was confirmed against a test plate; options bar, orientation row, selection, hit-test, inspector all exercised). 2D-mode only this build; the 3D-mode tile is the planned follow-on (the other half of the two-mode rule).

- **New HBS screw tile + tool.** The Draw-tab Members palette gains an **HBS** tile (screw icon) beside Bolts. In 2D mode it arms the `v25-screw` tool — single-click drops a `screw` entity at the cursor. The top options bar shows a **Size** select grouped by diameter (Ø8 TX40 / Ø10 TX40 / Ø12 TX50, labelled "Ø8 × 120") and a live 5-button **Orientation** row (plan + section each direction). Reuses the parked, ETA-verified catalogue and `screw` entity rather than reinventing them. (`js/72i-v25-screw.js` new; tile in `js/74-v26-bb-rail.js`; draw route / click / preview ghost / active-tile in `js/69-v25-dispatch.js`; options bar in `js/72-v25-options-bar.js`; icons + script tag in `index.html`.)
- **Drawn to scale, AS 1100 linework.** Section orientations draw the true side profile — pan head + under-head collar/taper, smooth shank, threaded sawtooth (crest at the thread major Ø standing proud of the shank, root at the minor Ø), pointed tip, and a dashed centreline — all sized from the catalogue (dK / dS / d2 / t1 / L / b). The plan (end-on) orientation reuses the existing head-circle + Torx-cross + clearance-ring drawer. (`js/72i-v25-screw.js` `drawScrew2D`; end-on via `js/77-screw-entity.js` `drawScrewEnt`; thread + polygon helpers from `js/33-draw-bolt.js`.)
- **Head subtly snaps to the outside face of a plate / steel member in section.** Placed in a section orientation, the screw head auto-lands on the nearest plate or member outer face along the screw axis (within a 400 mm window), with the threaded shank driving into the material; the click also catches plate/flange faces during placement. (`js/72i-v25-screw.js` `v25ScrewBearingFace`; `js/09-snap.js` per-axis face-snap tool list.)
- **Select / edit / export.** Click-selectable with a body grip; the inspector exposes Size (all 18 codes) and Orientation; DXF export emits the to-scale profile on the S-BOLT layer; save/load is automatic via `entities2D` JSON. (`js/71-v25-selection.js` bounds / handles / hit-test / inspector field block; `js/45-dxf-export.js` screw branch.)
- **Removed the leftover autoload demo.** `TMBR_AUTOLOAD_EXAMPLE` now defaults to `false` (the floating "Timber-screw example" button was already unwired in `index.html`); the Draw-tab tile is the proper discoverable surface. (`js/99-tmbr-autoload.js`.)

## 2026-06-03 — Weld lines: double-click popup no longer clashes; selection glow; smaller default hatch + per-weld size controls

Four auto-weld improvements (the diagonal fillet-weld hatching drawn along detected member↔member / plate↔member contacts, in both 3D and 2D modes). Browser-verified 2026-06-03 (app boots console-clean; `showWeldPopup` driven programmatically — the popup opens and sets the active-glow key, the three new size inputs show their defaults, a live edit writes into `weldOverrides`, an out-of-range value clamps to range, and close clears the popup + glow key). Built via a multi-agent workflow: 2 implementers + 4 adversarial reviewers (correctness / both-mode / regressions / structure), all passed.

- **Double-clicking a weld no longer opens the member end-cap dropdown on top of the Weld Properties popup.** When a weld ran along a member *end* (e.g. a cap/end plate welded to a column end), the double-click hit both the member end and the weld, and two separate `dblclick` listeners each opened a popup — the "Start end cap" dropdown landed over the weld popup and hid its **Enabled** (remove) checkbox. The first listener now bails out when the double-click lands on an auto-weld hatch or an SHS joint, leaving the dedicated second listener to own the weld/joint popup — so only the weld popup opens (and the misleading member selection no longer appears). Non-weld member-end double-clicks still open the end-cap dropdown; member-body double-clicks still open Settings. (`js/39-events.js` first 2D `dblclick` listener — guard via `v25HitTestWeld` / `hitTestJointV25`.)
- **The weld you double-clicked now glows subtle orange so it's obvious what you selected.** The auto-weld hatch is geometric (not a selectable entity), so there was no feedback on which weld the popup referred to. The interface whose popup is open now renders in orange (`#f59e42`) with a soft glow and slightly heavier ticks; opening the popup sets the active key and closing it clears it (re-rendering on both). Works in 3D and 2D — the 3D (`idA-idB`) and 2D (`v25-idA-idB`) key namespaces keep the highlight to the one clicked weld; `ctx.shadowBlur` is reset on both exits of `drawWeldHatch` so the glow can't bleed onto other linework. (`js/23-auto-weld.js` `weldPopupActiveKey`, `drawWeldHatch`, `showWeldPopup`, `closeWeldPopup`.)
- **Default weld hatch is slightly smaller and more closely spaced.** The diagonal ticks were a touch large; the defaults drop from 2.2 mm tick / 0.9 mm spacing to **1.8 mm tick / 0.7 mm spacing** (sheet-mm), reading cleaner at detail scale. (`js/23-auto-weld.js` `WELD_HATCH_DEFAULTS`, consumed in `drawWeldHatch`.)
- **New per-weld size controls in the Weld Properties popup: tick length, line thickness, spacing.** Double-click a weld → three numeric fields (clamped to sane ranges) override that weld's hatch length / line weight / tick spacing; edits apply live and persist with the drawing (stored inside `weldOverrides`, both modes, no save-schema change). `drawWeldHatch` now takes the interface object and reads these with fallback to `WELD_HATCH_DEFAULTS`; both call sites plus the 2D `computeV25WeldInterfaces` candidate carry the new fields. (`js/23-auto-weld.js` popup inputs + `apply()` + `computeWeldInterfaces`; `js/68-v25-tools.js` `computeV25WeldInterfaces` + `drawV25AutoWelds`.)

## 2026-06-03 — Grouped-plate-move undo fix + noteBox arrowhead pick priority

Two 2D-mode fixes. Browser-verified 2026-06-03 (hit-stack ordering + grouped-plate undo driven programmatically; app boots clean, console error-free).

- **Undo of a grouped assembly moved *by its plate* now reverts the whole group in one step.** Dragging a group by a v25 member already recorded one atomic `v25Move` undo, but dragging it by a **v2 plate** only committed the dragged plate's own transaction — the v25 mates (member/bolts/welds) and any sibling plates were moved with no undo record, so Ctrl+Z snapped just the grabbed plate back and left the rest displaced. A grouped plate move/rotate now joins the same `v25Move` group-undo mechanism: the dragged plate is direct-mutated (no separate v2 transaction) and the whole assembly — member + every plate — is captured in one undo entry, so one Ctrl+Z reverses it. **Non-grouped plate edits are unchanged** (still v2-authoritative, byte-identical). Joint welds/bolts created by a flange drop are still folded into the single undo. (`js/72f-v25-grouping.js` `snapshotGroup`/`plateGroupMoveUndoBegin`/`plateGroupMoveUndoCommit`; `js/v2/tools/edit-plate.js` body + rotate commit, drag-begin hooks, Escape reset. Reuses the existing `v25Move` restore in `js/05-state.js` — no change there.)
- **Clicking a noteBox arrowhead now selects the noteBox first, then click again for the member behind.** An annotation arrow points *at* a member, so its arrowhead almost always sits on top of one; pick priority was pure draw-order, so a member drawn after the noteBox stole the click. The arrowhead now wins the first click (an 8 px priority pass over leader tips), and **clicking the same spot again steps to the member underneath** (and wraps) — the Bluebeam/AutoCAD "click again to get the thing behind" behaviour. Because the arrow tip is also a draggable grip, a pure *click* cycles while a press-*drag* still re-aims the arrow tip (a 4 px dead-zone distinguishes the two). Moving >4 px, switching tool, or switching mode restarts the cycle. Leader-line picks, plain member picks, Shift multi-select, group-expansion and the double-click paths are unchanged. (`js/71-v25-selection.js` new `v25HitTestStack` + thin `v25HitTest` wrapper; `js/39-events.js` 2D select cycling + handle-grab click/drag deferral; `js/07-globals.js` `v25Cycle*` state; resets in `js/41-tools.js`, `js/69-v25-dispatch.js`, `js/65-v25-data-mode.js`.)

(`js/72e-v25-stiffener.js` `v25OpenStiffPopup`, wired into the 2D-mode double-click handler in `js/39-events.js`; **Steel hatch** toggle added to the stiffener options bar in `js/72-v25-options-bar.js`.)
- **Stiffeners only place on I-sections and channels.** The tool now hosts only on members with a clear top + bottom flange to stiffen between — **UB / UC / WB / PFC** (PFC added as a valid host) — keyed by member type with a section-name fallback. When you do tick it on, the ticks are 2–8 evenly-spaced, count-capped per edge, sized to the on-screen plate width (no more dotted-line smear when zoomed out), and the screen + DXF agree (weld off ⇒ no `S-WELD` ticks emitted either). The joint "Weld together" ticks (`drawJWeld2D`) keep the `LW.DIM` lineweight fix (was an undefined `LW.thin`). (`js/72e-v25-stiffener.js`, `js/72-v25-options-bar.js`, `js/72g-v25-joint.js`.)

## 2026-06-03 — Member hidden lines stay crisp at any zoom

- **Hidden-line dashes now scale with the sheet, not the screen.** The dashed lines for RHS/SHS inner walls, the UB hidden web, and PFC flange roots used the shared fixed-screen-pixel `DASH.HIDDEN` pattern, so at a normal detail zoom they collapsed into a faint near-solid dotted texture. They now use a sheet-mm dash (3 mm dash / 2 mm gap) scaled by `viewport.zoom`, so they read as proper hidden lines at any zoom **and** export true-to-paper (the vector-PDF path runs in sheet-mm with `viewport.zoom == 1`). ## 2026-06-02 — Bluebeam-style copy-drag (hold a modifier + drag any item to duplicate)

- **Hold Alt (or Ctrl on Windows) and drag any item to drop an exact copy.** Like Bluebeam's Ctrl-drag: grab the **body** of anything — a line, bolt, member, text box, hatch, leader, reo, a multi-selection, a whole group, a plate, or a 3D-mode object — drag, and a copy is created and follows the cursor while the original stays put. The copy is created on the **first drag movement** (a modifier-click that doesn't move makes no copy), and the copy is selected on release so you can immediately drag another. **One Ctrl+Z removes the copy** (plates take two — the place then the move). Grip / vertex / rotation / end-handle drags are unaffected (they still resize/edit); a plain drag still just moves. Works in both 2D paper-space and 3D model mode. (`js/07-globals.js` `isDupDragModifier` + `IS_MAC`; `js/39-events.js` v25 + 3D drag-start hooks; `js/71-v25-selection.js` `v25CloneEntsInPlace`; `js/v2/tools/edit-plate.js` plate copy-drag; `js/05-state.js` `v25Add`/`objAddMany` undo; `js/72f-v25-grouping.js` `v25NewGroupId`.)
  - **Modifier choice:** Alt/Option works on every platform (matches Figma/Illustrator); Ctrl additionally works on Windows, giving the exact Bluebeam muscle-memory. On macOS, Ctrl-click is deliberately left as the system right-click so the Group/Joint menu is untouched.
  - A duplicated **group** becomes its own independent group (fresh group id) — moving the original later no longer drags the copy. A duplicated **plate** is a fresh standalone plate (keeps its thickness/material, not its group).
  - The inspector's **Duplicate** button now shares the same clone routine and is undoable in one step (it previously left no undo entry).

## 2026-06-02 — Web stiffener actually renders (the "dotted line" bug) + STP-quality polish

- **Web stiffeners now draw.** Placing a stiffener (STIFF tile, 2D mode) showed only a faint dashed box and **no plate**: `drawStiff2D` / `v25PreviewStiffener` read `window.ctx`, but `ctx` is a top-level `let` in `07-globals.js`, so it is **not** a `window` property — `window.ctx` was always `undefined` and both functions hit `if (!context) return` before painting anything. The "dotted line" you saw was the **selection-highlight box** drawn (with the bare `ctx`) around the invisible, just-placed entity. Both now use the bare global `ctx` like every other drawer, so the plate and its placement ghost render. (`js/72e-v25-stiffener.js`.)
- **Stiffener lineweights fixed.** The renderer referenced `LW.medium` / `LW.fine` / `LW.thin`, which don't exist in the AS 1100 `LW` table (valid keys are `CUT/VIS_HEAVY/VIS/MW/DIM/HID/CL/HATCH`) — they resolved to `undefined`, so the intended weights were never applied. Outline → `LW.VIS`, steel hatch → `LW.HATCH`, weld ticks → `LW.DIM`. (`js/72e-v25-stiffener.js`.)
- **Stiffener reads as a clean plate at any zoom (STP 6011 quality).** A 10 mm plate is ~1 px wide at 1:10, so it was a ragged sliver. It now draws with a render-only **minimum on-screen width** (never collapsing below ~2 px) and a crisp `LW.VIS` outline, so a thin stiffener stays legible at working zoom without smearing to a black bar; the both-edges fillet weld is **2–8 regular capped ticks** (drawn only when there's room) instead of a dotted line. Geometry is unchanged — the plate still spans exactly the UB **inner flange faces** (`v ± d/2 ∓ tf`); verified flush top and bottom. (`js/72e-v25-stiffener.js`.)
- **Steel cross-hatch is opt-in per stiffener.** By default a stiffener is an open outlined plate (clearest at small scale — no auto solid-fill). Tick **Steel hatch** to fill it with AS 1100 45° steel cross-hatch — either on the Stiffener **quick-options bar** before placing (alongside Thickness 6–20 mm and Weld-both sides), or per-stiffener by **double-clicking** it to open a small properties popup (Thickness / Weld both sides / Steel hatch). The choice persists to the next placement. (`js/72e-v25-stiffener.js`, options bar `js/72-v25-options-bar.js`, dblclick wired in `js/39-events.js`.)
- **Joint weld ticks now draw too.** `drawJWeld2D` (the joint "Weld together" fillet ticks) carried the identical `window.ctx` early-return + `LW.thin` bug, so a welded joint also showed nothing; fixed the same way (bare `ctx`, `LW.DIM`). (`js/72g-v25-joint.js`.)

## 2026-06-02 — Plate corner-resize + marquee multi-select

Matches the v25-member marquee (Shift adds to the selection, plain replaces). (`js/v2/tools/edit-plate.js` `selectInRect`.)

## 2026-06-02 — Plate drag rework, subtle fill, multi-member grouping

Follow-up to the edge-drag from earlier today (which over-grabbed thin plates and skewed them). Reworked to match how Dan actually edits plates.

- **Plate drag, corrected.** Clicking a plate's **body or a long edge and dragging now moves the whole plate** (no more skewing). Grabbing a **corner** and dragging now moves that whole **END edge** — both corners of the short edge together, constrained to the plate's long axis — so the plate widens/narrows and **stays rectangular** (works for horizontal and vertical plates, any orientation). (`js/v2/tools/edit-plate.js`, previews in `js/v2/ui/live-render.js`.)
- **Selected-plate fill is now a light wash.** Plates were filling **opaque dark orange** when selected because the theme accent is an `oklch()` colour and `colorAlpha()` couldn't apply alpha to oklch (so the 14% fill rendered at 100%). `colorAlpha()` now rasterises wide-gamut colours to sRGB before applying alpha (`js/08-coords.js`), so the selection fill is the intended subtle ~14% orange — and **every other translucent fill** (drag guides, etc.) renders correctly in the oklch themes too.
- **Group any number of members, plates included.** Grouping already handled multiple v25 members; now **multiple plates** can be co-selected and grouped as well. **Shift+click** a plate toggles it into the selection (the universal multi-select convention, matching v25 members), and `Ctrl+G` groups everything selected — members and plates, **touching or not** — into one unit that **moves and rotates together**. Re-grouping merges existing groups instead of fragmenting them. (`js/72f-v25-grouping.js`, `js/v2/tools/edit-plate.js`, `js/v2/ui/live-render.js`.) *(Trade-off: Shift+click is now multi-select, so the old Shift+click plate-vertex insert/delete is retired — rectangular plates don't need it; can be restored on a different gesture if wanted.)*

## 2026-06-02 — Joint-menu fixes, plate edge-drag, flash pulse

A follow-up round on the grouping/joint feature (diagnosed with a multi-agent investigation workflow).

- **2D mode: moving a v25 entity (member/hatch/blockwork/leader/noteBox/grouped assembly) is now undoable; Ctrl+Z no longer deletes a recently-placed plate/column instead of reversing the move.** Dragging a v25 entity to move it now records exactly one atomic `v25Move` undo action capturing the before/after of the entity (and, for a grouped assembly, every group mate incl. v2 plate polygons), restored by a new branch in base `undo()`/`redo()` (`js/05-state.js`, `js/39-events.js`). *(Out of scope: a grouped move initiated by grabbing the plate body itself still leaves grouped v25 mates unrecorded — needs the v2/grouping path.)*

The flash now applies its opacity via `ctx.globalAlpha`, so it's a true translucent wash in every theme (`js/72f-v25-grouping.js`). *(Note: the same `colorAlpha`+oklch limitation makes other translucent fills — e.g. the plate-selection fill — render opaque in the oklch themes; not changed here, flag for a separate pass if you want those restored.)*

Hold **Shift** to move only the nearest node; **Shift+click** (no drag) still inserts a vertex, **Shift+click on a corner** still deletes it (`js/v2/tools/edit-plate.js`, preview in `js/v2/ui/live-render.js`). - **Plate-on-beam now defaults to *No connection* and opens the menu.** Dropping a grouped end-plate onto a beam flange no longer auto-welds it. The geometric auto-weld is suppressed for the plate↔beam interface the joint owns (member-scoped, so the plate↔column cap weld you liked is preserved), the joint defaults to **no connection**, and the connection menu now **auto-opens** at the contact line so you can pick Weld / Bolt / None (`js/68-v25-tools.js`, `js/72g-v25-joint.js`).
- **Joint menu is readable and its actions work.** The connection dropdown was dark-on-dark (text invisible until hover) because it used an undefined `--panel-bg` token; it now uses the defined `--surface-2` surface + `--text` (readable in all five themes), with a contrasting ink on hover. Clicking **Bolt together** / **No connection** / **Weld together** now actually applies — the menu's outside-close listener was a capture-phase `pointerdown` that tore the menu out of the DOM *before* the item's `mousedown` could fire; it's now a guarded capture `mousedown` that ignores in-menu clicks. The same readability + close-listener fix was applied to the Group/Ungroup context menu (`js/72f-v25-grouping.js`).
- **Bolt label fixed to 2/M20.** The bolt option read "2/M50" (it was concatenating the 50 mm edge distance); it now reads **"Bolt together (2/M20)"**, matching the M20 bolts actually created (`js/72g-v25-joint.js`).

## 2026-06-02 — Grouped members: group-flash feedback + rotate-together

- **Grouping now shows a confirmation flash.** When you `Ctrl+G` a multi-selection (e.g. a steel column + an end plate), every member of the new group is briefly washed in the accent colour, fading out over ~3 s, so the grouping reads as a deliberate event rather than a silent state change. The wash paints the true member outlines (and the plate polygon), is self-driven via `requestAnimationFrame`, and is keyed off the group id — so it persists for the full fade even if you immediately click away. New `startGroupFlash` / `drawGroupFlash` in `js/72f-v25-grouping.js`, painted by the v2 live-render wrapper (`js/v2/ui/live-render.js`) after the plate + bolt bodies so it sits on top of both the v25 members and the v2 plates.
- **Grouped members now rotate together.** Rotating any member of a group rotates the whole assembly rigidly about a single pivot — so a column + cap-plate group can be flipped 180° to turn a cap plate into a base plate in one move. Both rotation entry points are group-aware: the v25 member rotate-ball (column / blockwork strip / hatch — `js/71-v25-selection.js`, pivot = that member's own midpoint/centroid) and the v2 plate rotation handle (`js/v2/tools/edit-plate.js`, pivot = the plate centroid). The mate-rotation mirrors the existing move-together design — incremental for the v25 path, snapshot-based for the plate path — and **Escape mid-rotate now restores the whole group** to its pre-drag pose (true for cancelled grouped moves too). v2 plates rotate every polygon vertex; stiffeners rotate both endpoints; members rotate their start point and advance `rot` by the same delta (a rigid-body rotation). 2D-mode only.

## 2026-06-01 — 2D-mode plate: marquee selection + Ctrl+Z undo/redo

- **Plates now obey marquee (drag-box) selection.** Dragging a selection box around a 2D-mode (v2) plate now selects it — window-box (L→R) when the plate is fully enclosed, crossing-box (R→L) when it's merely touched — matching every other member. The v2 edit-plate tool gains `selectInRect()`, which the v1 marquee (`js/39-events.js`) calls after its own `entities2D` pass; read-only plate mirrors (`_v2Mirror`) are excluded from the v1 hit list so they're never double-counted. Root cause was a variable-shadowing bug: inside the marquee handler `v2` is a local (the box's max-V coordinate, `const v1 = …, v2 = Math.max(…)`), so the wiring's `v2.tools.editPlate` read a *number's* property and silently no-op'd — fixed by reaching the namespace via `window.v2`. (Limit: the v2 tool holds one selection, so a box over several plates selects the topmost.)

Canvas clicks pick endpoint members in order (Priority 1 first, then 2, 3, ...); each click writes a partial `priorityOrder` so the cut math updates live as the user picks. Enter / popup "Apply" commits; Esc / outside-click reverts to the snapshot. Auto-commits when every endpoint has been picked.

For the joint currently being edited, strokes each endpoint member's bounding outline in `--accent` colour at `LW.CUT × 1.5`, with `globalAlpha` oscillating 0.4 ↔ 1.0 at 1.5 Hz (sine driven by `performance.now()`). The currently-locked Priority 1 (first picked member) gets a thicker static outline with full alpha so the user can tell it's committed. A `requestAnimationFrame` loop in the popup module pumps `requestRender` while pick mode is live.
- N-way priority data model in `v25JointMeta[viewKey][jointKey].priorityOrder = [memberId1, memberId2, …]`. This is now the SOURCE-OF-TRUTH for joint priority; `v25JointPriority1(joint, viewKey)` reads it first and falls back to the legacy `priorityForPairV25` map. `v25SetJointPriorityOrder` rebuilds the pair map from the ordered list so the existing cut math in `_computeEndCutV25` keeps working unchanged: only Priority-1 ↔ other-member pairs get winner entries; all other pairs are cleared (matches the brief's "everyone clips to Priority 1" rule). `v25SetJointMode('mitre')` clears both `priorityOrder` and the pair entries. `v25JointMembersInOrder` is the canonical lookup the popup uses to render the ordered list.

Replaces the legacy "Auto-mitre joins" block whenever the selected `mem2` participates in a `computeShsJointsV25`-detected joint in the active view. New section shows: member count, mode (Mitre / Priority — with "auto, N-way" label when 3+-way locked), the selected member's role at the joint (end A / end B / through-chord), the selected member's priority position ("P2 of 3"), and an "Edit joint…" button that opens the full joint popup anchored at the button's screen position. Two small renderer extensions: a new `'btn'` field kind for full-width action buttons, and a `button[data-action]` wire-up that currently dispatches `'openJointPopup'` (extensible — add new actions here as later gates need them). Inspector and popup are now equivalent entry points: changes from either route update the canvas and the open popup live.
- Drag-sync of joint metadata (Gate 7 / Joints Phase 5). New `v25SweepJointMeta(viewKey)` in `23a-shs-joints.js` drops `v25JointMeta` entries whose joint keys no longer match a live joint from the detector. Called from the V25 drag-end (mouseup) handler in `39-events.js` after `v25Drag = null`, so a drag that pulls a member's end out of its joint cluster cleans up orphan metadata instead of accumulating dead entries. Idempotent. Sweeps a single viewKey at a time (passed from `activeBlock.viewKey`). The legacy `priorityForPairV25` pair map is intentionally NOT swept — pair entries become inert when their members separate and auto-reactivate if the user drags them back together later, preserving the priority intent across small drag corrections. For 3-way joints where one member drags away, the surviving 2-way joint starts with empty metadata (true priority-order migration is a Gate 8 polish item).

Called from `exportSheetToDXF` after the 2D entity emission pass, scoped to non-isometric non-hidden blocks. - Joint mutation undo coverage (Gate 8 / Joints Phase 6). All three joint setters (`v25SetJointCapPlate`, `v25SetJointPriorityOrder`, `v25SetJointMode`) now capture before/after snapshots and register a `v25JointMeta` undo entry. Snapshot includes both the `v25JointMeta[viewKey][jointKey]` record AND the `priorityForPairV25` entries derivable from the joint key. New helper `_v25ApplyJointSnapshot(snap)` does the full restore (clears pair entries set by the mutation, re-applies the snapshot's, invalidates caches, refreshes the inspector and popup). New `act: 'v25JointMeta'` handler in `05-state.js`'s `undo` and `redo` dispatchers routes to the restore helper. Stale-undo-suppression: identical before/after snapshots are skipped (no-op mutations don't pollute the undo stack).

- 3-way → 2-way joint priority migration (Gate 8). When `v25SweepJointMeta` finds an orphan joint key whose endpoint set is a STRICT SUPERSET of some live joint's endpoint set (e.g., a 3-way joint lost one member to a drag and is now 2-way), it MIGRATES the orphan's metadata to the surviving sub-joint. The migrated `priorityOrder` is filtered to keep only the surviving members in their relative order, so a `[A, B, C]` priority order on a 3-way joint becomes `[B, C]` on the surviving 2-way joint if A dragged away. `priorityForPairV25` entries are re-derived from the migrated `priorityOrder` so the cut math sees the new Priority 1 without a separate setter call. Won't overwrite a sub-joint that already has its own metadata.
- Live refresh inspector ↔ popup (Gate 8). New `v25RefreshJointPopupIfOpen(viewKey, jointKey)` in `75-v25-joint-popup.js` re-renders the open popup if it's for the joint being mutated. The notifier skips refreshing the SOURCE UI to preserve input focus during typing, but refreshes the other side. Undo/redo (no source) refreshes both.

## [V25.modular] — 2026-05-02

The structural refactor. Pure file split, **zero behaviour change**, verified by browser smoke test.

### Changed

- The single 22,241-line `index.html` is now a 1,308-line thin shell + `css/styles.css` (~1,500 lines) + 73 numbered `js/NN-name.js` files (~19,800 lines total, average ~270 per file).

- All scripts loaded as classic `<script>` tags (NOT `type="module"`) — globals continue to flow between files exactly as in the monolithic original.
- New developer playbook (`CLAUDE.md`) + project README + this changelog.
- `HANDOFF_V2.md` (which described an abandoned Bluebeam-shell rewrite) moved to `archive/handoff_v2_abandoned.md`.
- Pre-existing backup files (`index.html.pre-v25-backup`, `dev/index.html.pre-layout-overhaul-backup`) moved to `archive/snapshots/` with date-prefixed names.

### Verification

- All 73 JS files pass `node --check`.
- All 20 critical globals (canvas, ctx, viewport, blocks, tool, project, V25 / V14 / V22 fns) defined exactly once and reachable from the global scope.
- Browser smoke test: app loads, renders A1 sheet, places objects, switches 2D ↔ 3D mode, places V25 anchor entity, runs `render()`, runs `updateStatus()`, runs `v3dRebuildScene()`, multi-sheet add/switch, save/load JSON round-trip — **all clean, no console errors, no warnings**.

### Known carryovers (NOT fixed in this PR)

These are pre-existing in the source. The refactor preserved behaviour exactly; fixes belong to their own PRs. See `CLAUDE.md → Known issues`.

1. Duplicate `function v25Mem2Thickness(ent)` in `68-v25-tools.js`.

### Structural impact (why this matters for product longevity)

- A change in any one tool now touches one file (typically 100–700 lines), not a 22k-line monolith.
- Permission prompts in chat-driven dev are fewer because target ranges are unique.
- Adding a feature has a clear home rather than appending to the bottom of a giant file as another `// V<n>` section.
- The next round of work (Phase 2 in CLAUDE.md) — break up `initEvents`, lift globals into `appState`, replace monkey patches with extension hooks — becomes safe and reviewable in isolation.
