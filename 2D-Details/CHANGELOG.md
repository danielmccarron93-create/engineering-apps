# Changelog

All notable changes to StructDraw from V25.modular onwards. Pre-V25 history (V0 → V24.A3) predates this changelog and was not migrated into the repo.

The format is loosely based on [Keep a Changelog](https://keepachangelog.com/), but pragmatic — entries are by phase / dev session, not by semver.

## 2026-06-05 — ChemSet anchor stud: selection polish + editable embedment depth

Usability pass on the 2D-mode ChemSet anchor stud (`stud`). All five behaviours below, browser-verified.

- **Selection now reads like every other member.** Selecting a stud gives a subtle orange infill that **hugs the glyph** instead of a box floating off to one side. The old highlight was built around the raw click point and ignored the bearing-face snap that re-centres the drawn rod, so once the washer snapped to a plate/member face the box drifted by tens of mm. The highlight (and the hover halo) are now built from the same bearing-recentred centreline the glyph uses. The **same fix is applied to the HBS screw and the AS 1252 bolt**, which shared the identical offset. (`js/71-v25-selection.js`.)
- **Alt-drag duplicates a stud**, exactly like every other member (select, hold **Alt**, drag the body → an independent copy carrying its size/orientation/embedment). *Note:* the duplicate modifier on macOS is **Alt**, not Ctrl — Ctrl-click is a system right-click and Ctrl is the zoom modifier, so it can't be used for drag-duplicate; this matches all members and needed no stud-specific code.
- **Top options-bar dropdown for a selected stud.** With the Select tool and one stud picked, the bar shows **Size · Orientation (icon row) · Embedment (mm)** — the same surface the other fixings get for placement, now for editing a placed stud. (`js/72-v25-options-bar.js`.)
- **Drag-edit the embedment.** Two new grips on a section stud: **drag the chisel tip** to set the embedment depth (in clean **5 mm steps**, with the depth number ticking **live** in both the top bar and the inspector as you drag, so you can just drag until you hit the value you want), and **drag the drilling datum** (the round node where embedment = 0) to re-datum it — it **snaps to host faces** (top of grout, top of blockwork, a plate underside…) so you can drop the drilling start from the grout down to the blockwork; the depth (and tip) ride down with it. The drag never moves the stud's placement. (`js/71-v25-selection.js`, `js/72j-v25-stud.js`.)
- **Two-tone read about the drilling datum.** The rod is now shaded in two zones split at the datum node: **solid dark steel above** (the head + the shaft passing through the grout/fixture — *not* drilled) and **light fill with the adhesive hatch below** (the drill hole + epoxy bond — the depth the builder actually drills). A crisp datum line marks the node. So re-datuming to the top of blockwork correctly shows the grout as steel and the embedment as starting at the blockwork. (`js/72j-v25-stud.js`.)
- **Type the embedment depth.** A numeric **Embedment depth (mm)** field in both the inspector (Settings) and the top bar — type e.g. `300` and the rod extends **live as you type** so it's drilled 300 mm **from the datum** (the grout above the datum is excluded — what the builder needs). Typing in the bar no longer rebuilds it mid-keystroke, so you can enter the full number without losing focus, and the two fields stay in sync. An **Edge offset (mm)** field exposes the datum position too. The shown defaults are the values actually drawn (WYSIWYG); clearing a field reverts to the catalogue value. (`js/71-v25-selection.js`, `js/72-v25-options-bar.js`, `js/72j-v25-stud.js`.)
- **Zero change to existing drawings, DXF kept in sync.** Geometry is unified through one helper (`studSectionGeom`) read by the drawer, DXF, hit-test, hover, footprint and grips, so they can never drift. With no embedment override set, every stud renders **byte-identical to before** — for all sizes M8–M24 *and* whether or not it's snapped to a host (the default embedment is back-computed from the catalogue length, so the dual-embed M20/M24 rows don't shift). A typed/dragged depth now also exports true to **DXF**. (`js/72j-v25-stud.js`, `js/45-dxf-export.js`.)

## 2026-06-05 — Welded truss joints: per-member weld priority (N>2 members)

The 2D-mode member-to-member auto-weld/joint engine now handles **truss nodes where more than two members meet**, with a per-member **weld priority** that the engineer controls directly.

- **Priority cascade at every joint.** Each 2D member (`mem2`) carries an optional integer **weld priority** (1 = highest = the *solid* member that runs through; lower number wins). At a node, a member is cut-and-welded to **every neighbour that out-ranks it**, and a **through-chord always cuts** (a member can never run solid *through* a chord that crosses it). So a chord (priority 1) runs solid, the first diagonal is cut to the chord, and a second diagonal meeting the first is cut to *both* — exactly the truss cascade. **Default priority follows draw order** (first-drawn = 1), so a freshly-drawn truss cascades correctly with no clicks. (`js/23a-shs-joints.js`.)
- **Double-click a welded member → a weld-priority dropdown.** Double-click a welded member's body (or its joint node) and a popup opens with a one-click **"Run through (make solid)"**, a **`Mitre, 1, 2 … N`** dropdown, and a live list of every member in the welded group badged **SOLID / MITRE / CUT** so the resolved joint is always visible. Picking a number re-ranks via **insert-and-shift** across the connected group (the rest shift down), as a single **atomic undo**. The same "Rank" control is in the member's inspector (Settings). (`js/68-v25-tools.js` popup, `js/39-events.js` dblclick, `js/71-v25-selection.js` inspector, `js/05-state.js` undo.)
- **Plain 2-member corners still default to mitre** (portal-knee behaviour unchanged), and **older drawings render identically** — a joint that already carries an explicit priority/mitre choice keeps its exact look; only previously-untrimmed 3+ member nodes start auto-cascading by draw order.
- **Multi-cutter cut faces now draw correctly.** When a member is cut by two others at once (e.g. the third diagonal nestling against both the chord and the second diagonal), its end is drawn as a **kinked cap** that follows both faces, with the AS 1101 fillet-weld hatch tracking each segment — instead of a straight cap that mis-drew the member by ~20 mm at the notch. (`js/68-v25-tools.js` `drawMem2D` SHS/RHS/CHS · UB/UC · PFC · flange-face branches.)
- **Known gaps (deferred, documented):** DXF still exports the *untrimmed* member outline (PDF is correct); the weld is shown as the AS 1101.3 fillet **hatch** (a leadered weld symbol with leg size is a separate enhancement); 3D-mode joints keep their existing size-based priority popup. 2D-mode only.

## 2026-06-05 — Page navigation: Shift+scroll fix for physical mice + bottom-ribbon page stepper

- **Shift+scroll now flips pages on a physical mouse.** Holding Shift makes most mice/OSes deliver the wheel as *horizontal* scroll (`deltaX`), so the page-flip — which only watched `deltaY` — did nothing on a real mouse (it had only ever worked with trackpad/synthetic `deltaY`). It now accumulates whichever wheel axis dominates, so Shift+scroll-down → next page and Shift+scroll-up → previous, on a mouse and a trackpad alike. (`js/39-events.js`.)
- **A page stepper in the bottom status bar: `‹ N / total ›`.** Click `›` for the next page, `‹` for the previous; the label shows the current page and total, and the arrows grey out at the first/last page. Sits in the bottom ribbon and stays in sync with Shift+scroll and thumbnail clicks (driven by `renderPagesTab`, which every page/file change funnels through). (`index.html`, `js/49a-file-tabs.js`, `js/74-v26-bb-rail.js`, `css/styles.css`.)

## 2026-06-05 — Multi-file workspace: fast page-flip + 2-column page-thumbnail rail (polish)

Two refinements for working through multi-page documents (imported PDFs especially).

- **Shift+scroll now flips pages proportionally to how hard you scroll — snappy, minimal lag.** A gentle notch steps ~1 page; a fast flick flies through several and lands once, jumping straight to the target page in a single load (only the page you land on rasterises — the skipped ones don't). The on-screen PDF bitmap cache was widened (8 → 24 pages) so flipping back and forth through a ~20-page document redraws each page instantly from cache, and a page-flip now just moves the active highlight in the rail instead of rebuilding it. (`js/39-events.js`, `js/83-pdf-document.js`, `js/07-globals.js`, `js/74-v26-bb-rail.js`.)
- **The left Pages tab is now a 2-column grid of page thumbnails.** Instead of a single column of text rows, each page shows a small rectangular preview of its content with the page number + size tag — sized so many fit, so you can scan a long document at a glance and click straight to the page you want. PDF pages preview via pdf.js; native detail pages preview the active page live (others fill in as you visit them). Thumbnails generate lazily, throttled (2 at a time), and cache per page (file-scoped, so two open files never cross-contaminate) so switching never regenerates or flickers; an edited page's thumbnail refreshes. (`js/85-page-thumbnails.js` NEW, `js/74-v26-bb-rail.js`, `js/04-workspace.js`, `css/styles.css`.)

## 2026-06-05 — Member end-grip drag: Shift = 45° extend; default grid 5 mm

Two small drawing-feel updates.

- **Hold Shift while dragging a member's end grip → the member snaps to the nearest 45° and you just extend it.** Grab either end of a 2D-mode member (`mem2`) with the Select tool and drag; press and hold Shift and the dragged end locks onto the nearest **45° line** (0 / 45 / 90 / 135 / …) through the **fixed** end — whichever eighth-angle the drag is closest to — so the member becomes exactly horizontal, vertical, *or* a clean diagonal (e.g. a 1:1 brace) and you extend it straight along that direction. At 0/90/180/270 it behaves exactly like the previous horizontal/vertical lock. Release Shift for a free angle. Works for **every member type** (UB/UC/SHS/RHS/PFC/CHS/EA/UA/timber) — they all share one `mem2` end-grip path. Complements the existing *body*-drag ortho lock (2026-06-04), which deliberately skipped grip handles; this adds the same feel to end-grip extend. Implemented as a 45° angle projection (project the dragged end onto the closest eighth-angle line through the fixed end) in the `mem2` `end-a`/`end-b` branch of `v25Move`, gated on the live `shiftHeld` global, so the recomputed length/rotation and the auto-mitre host probe all key off the snapped position. (`js/71-v25-selection.js`.)
- **Default grid spacing is now 5 mm (was 10 mm).** The finer grid gives a nicer placement/snap feel out of the box; the `Grid` dropdown still offers 1 / 5 / 10 / 25 / 100. (`js/07-globals.js` default `gridSize`; `index.html` dropdown default.)

## 2026-06-05 — Multi-file workspace: multiple files open at once + PDF import (Phases 1–4)

The headline build: StructDraw goes from one open project to **several open files at once**, plus first-class **PDF import** for marking up other people's drawings. Built across three reviewed + browser-verified passes.

- **Open multiple files as top-bar tabs.** A row of file tabs sits in the top bar (name · unsaved-dot · ×); the **+** opens *New file / Open .sdproj… / Import PDF…*. Each file is its own set of pages (what used to be the single "project"); the live `project` global is repointed at the active file, so every existing tool, the render loop, and the left **Pages** rail work unchanged on whichever file is active. Click a tab to switch (the whole drawing + the Pages rail swap, with full isolation); double-click to rename; × to close (you can never close to zero — the last close opens a fresh empty file). (`js/04-workspace.js` NEW, `js/49a-file-tabs.js` NEW, `js/50-project.js`, `js/62-toolbar.js`, `js/73-init.js`, `index.html`, `css/styles.css`.)
- **A brand-new file shows a landing** — *New blank A1 page · Open saved file… · Import PDF* — instead of a blank canvas.
- **Per-page size.** Pages now carry their own size/orientation (`activePageSize()`); native detail pages default to A1 (byte-identical to before), imported pages take the PDF's real size. Render, fit-to-view, layout, and PDF export are all page-size-aware. (`js/01-config.js`, `js/05-state.js`, `js/22-render-core.js`, `js/63-layout.js`.)
- **Auto-restore session.** Every open file/page is continuously saved to the browser's IndexedDB (debounced, structured-clone so embedded PDF bytes survive), and restored on relaunch — close the tab and come back to exactly where you were. Per-file `.sdproj` save still works for sharing/archiving, and now embeds any imported PDF as base64 so the file stays self-contained. A small dot on a tab marks unsaved-to-disk changes. (`js/84-session-store.js` NEW, `js/04`/`05`/`50`.)
- **Import a PDF as a new file.** *Import PDF…* (top **+**, the File menu, or the empty-file landing) loads a PDF via pdf.js into a new file named after it, with **one page per PDF page** at the PDF's true size/orientation and **no StructDraw title block** (it's someone else's drawing) — listed in the Pages rail as "Page 1 · A3 · 2D mode" etc. The full 2D toolset (dimensions, notes, leaders, clouds, members, hatches) works on top. **Shift+scroll** moves between pages; plain scroll still zooms. (`js/83-pdf-document.js` NEW, pdf.js 3.11.174 CDN, `js/07`/`39`/`49a`/`62`/`74`/`index.html`.)
- **The imported PDF renders crisp at any zoom.** `drawPdfBackground` rasterises the active PDF page to a device-resolution offscreen bitmap, cached per zoom-bucket (LRU-capped), drawn beneath your markup and re-rendered sharply when you zoom in — so you can zoom into rebar/bolt detail and the PDF stays sharp. Async (a brief "rendering…" hint, never a blocked frame); the PDF page's intrinsic rotation is honoured. Native pages skip it entirely. (`js/83-pdf-document.js`, hook in `js/22-render-core.js`.)

## 2026-06-05 — Multi-file workspace: per-page-size PDF export + PDF-background raster (Phase 5)

- **PDF export now sizes each output page to that page's own dimensions and orientation.** The single-sheet (`exportSheetToPDF`) and project (`exportProjectToPDF`) exporters route the jsPDF page format through a new `_pdfPageFormat(page)` helper (page `size` → `[w,h]` + landscape/portrait), instead of the hardcoded A1 `[841,594]` landscape. A mixed file of native A1 + imported A3 + A4-portrait pages now exports each page at its true size. Native A1 pages resolve to `[841,594]/'landscape'` — byte-identical to before. (`js/44-pdf-export.js`, `js/51-multi-page-pdf.js`.)
- **A PDF-background page renders into the export crisp (~300 DPI) under the markup.** For a page whose `bg.type==='pdf'`, the exporter pre-rasterises the source PDF page to a ~300 DPI canvas via `window.renderPdfPageToCanvas(file, bg, targetPx)` and hands it to a new export override in `js/83-pdf-document.js` (`window.setPdfExportRaster`); `drawPdfBackground` then draws that bitmap into the page rect during the export render pass — after the white page fill, under the vector overlay — so the consultant PDF prints sharp with your dimensions/notes/leaders on top. The override bypasses the on-screen zoom-bucket cache (the export render runs at `viewport.zoom=1`, where the live bucket would pick a coarse raster) and is always cleared after each page so it never leaks into the next. (`js/83-pdf-document.js`, `js/44-pdf-export.js`, `js/51-multi-page-pdf.js`.)
- **Title block stays suppressed for imported PDF pages on export** (it already was on screen): `render()`'s existing `pageHasTitleBlock` gate drives both, and the multi-page exporter now sets `project.activeSheetIdx` per page so the page-aware render helpers (page rect, title-block suppression, PDF background) all key off the page actually being rendered — previously every exported page was drawn at the original active page's size, a latent bug for multi-size files.
- The exporters are now `async` (raster staging awaits pdf.js); all callers are fire-and-forget UI handlers, so behaviour is unchanged. PDF-background pages embed a lossless PNG of the raster (crisp; larger file) — a JPEG-underlay size optimisation is a possible follow-up.

## 2026-06-05 — Drag-snap polish: fixing→fixing alignment connector + tightened magnet

Final polish on the 2D-mode drag-snap (the Shift-ortho + fixing alignment magnet shipped 2026-06-04). All three are subtle, on-screen-only refinements — no change to placed geometry, export, or grip / vertex / rotation editing.

- **Faint orange alignment connector between two fixings.** When a dragged bolt / screw / stud's centreline catches onto *another fixing's* centreline, a fine-dotted orange (`#f29f1d`, alpha 0.5, `[2,3]` dash, 1 device-px floor) segment now draws **bounded between the two fixings** — a Revit/Bluebeam-style "these two are aligned" hint — instead of the loud full-canvas blue guide. The full guide is still used for fixing→member-edge and member→member snaps. Fixing↔member and member↔member snaps carry no `srcId`, so they never draw a connector. (`js/68-v25-tools.js` `edgesFor`/`_mkSnap`; `js/12-edge-snap.js` `drawEdgeSnapLines`.)
- **No redundant caption on the connector.** The floating snap label (which names a caught face, e.g. "UB flange inner") is now suppressed for fixing→fixing connectors — the orange line is self-explanatory and a blue "BOLT centre" caption beside it clashed in colour and added clutter against the STP 6011 bar. The label robustly picks the first non-connector snap, so a co-occurring member-edge guide still gets its caption. (`js/12-edge-snap.js`.)
- **Tighter, zoom-invariant magnet + no Shift-engage jump.** Catch/break tightened to ≈3 px / ≈5 px (was 5/9; px-based so zoom-invariant) for an easier pull-through. On a Shift-ortho engage the soft-snap correction is now folded into the drag baseline so the item lands exactly on its original start line instead of jumping by the snap offset. Soft-snap state also resets cleanly at the drag-arm sites and on Shift-engage. (`js/68-v25-tools.js` `v25ApplySnap`; `js/39-events.js` drag handler.)
- **Connector exact on a simultaneous two-axis catch.** When a dragged fixing catches on *both* its u- and v-centrelines at once, the connector's dragged endpoint is now re-read after both axes resolve, so its near end always sits on the fixing's final centre (previously the u-connector, built before the v-snap applied, floated up to a few px off in v). The normal single-axis catch was already exact; this just extends that to the rare both-axes case. (`js/68-v25-tools.js` `v25ApplySnap` post-apply reconcile.)
- **Grouped assemblies no longer tear on a snap catch.** The alignment soft-snap is now skipped while body-dragging a *grouped* item (a welded column + end-plate, etc.). `v25Move` translates the whole group rigidly, but the soft-snap was handed only the grabbed entity — so a catch shifted that one member (and its `pts[]`) off its mates by the snap offset, a tear that persisted after drop. A group now slides as one unit; the magnet still works on ungrouped members and fixings exactly as before. (`js/39-events.js` drag handler — added a `groupId` guard.)

## 2026-06-04 — Drag: hold Shift mid-move for ortho lock + subtle alignment snap for fixings

Two refinements to how an already-placed 2D-mode item moves when you click-drag it. The existing free drag stays the default — both only add help on top, and neither touches grip / vertex / rotation editing.

- **Hold Shift *after* you start dragging → orthogonal lock from the start point.** Drag a member / bolt / screw / stud freely as before; press and hold Shift mid-drag and the move locks to a straight horizontal **or** vertical line measured from where the item started (its original "fixed existing" line). The locked axis follows whichever way you're currently dragging — swing from mostly-sideways to mostly-down and it flips, snapping the other axis back onto the start coordinate — so a bolt nudged straight down stays exactly in its original vertical plane. Release Shift and it's free again. The hand-off is seamless: if the soft-snap below has nudged the item onto a neighbour mid-drag, that correction is folded into the drag baseline, so engaging Shift locks from the item's *true* current position rather than jumping by the snap offset. Reuses the existing `v25OrthoSnap()` (until now only wired into the draw tools): the body-drag arm-sites record the start point (`startU/startV` on `v25Drag`) and the mousemove drag handler applies the constraint. Body drags only — grip / vertex / rotation handles keep precise editing. (`js/39-events.js`.)
- **Subtle alignment snap for fixings (and now zoom-independent for members).** While dragging a bolt / screw / stud, when its centreline lines up with another fixing's centreline — or a member's edge / centreline — it gives a gentle magnetic catch so fixings land co-planar (the common "this bolt shares the same vertical line as that one" case). A soft catch/break zone (≈3 px catch, ≈5 px break — a 2 px hysteresis) lets you pull straight through to place freely. The catch tolerance is screen-px based rather than a fixed 2 mm, so the magnet feels the same at any zoom — the old 2 mm was sub-pixel at normal detail zoom and effectively never caught. Feedback: snapping onto a **member edge / centreline** draws the usual faint full-canvas guide; snapping onto **another fixing** instead draws a faint dotted **orange** connector bounded between the two fixings (a Bluebeam-style "these two are aligned" hint — see the 2026-06-05 polish entry). A dragged member still snaps to other members only (it ignores fixings, by design). Skipped while Shift ortho-locks, so Shift gives a pure straight slide. (`js/68-v25-tools.js` — `v25ApplySnap` extended to `bolt2` / `screw` / `stud`.)

- **Timber‑screw connection feature — Phase 4: rule engine (`dev/` only) — v1 acceptance gate MET**. New file `79-checks-timber.js` (~440 lines) implementing the pure-function `checkConnection(connection, model?) → ConnectionCheckResult`. Algorithm walks the nine steps from `PlannedBuilds/timber-screws/03-rule-engine.md` exactly: (1) resolve rule context — picks the right `RuleSet` and `CapacityTable` from the catalogue based on `connectionType` × substrate family × `preDrilled`, looks up `k_mod` from EN 1995-1-1 Table 3.1 via service class × load duration, looks up `k_dens` from the rear member's `TimberClass`, fails fast with a clear `failingItems` entry if any lookup misses; (2) build geometric reference frame — normalises `rear.grainDir` to unit `grain`, derives `perp` as 90° CCW, normalises `connection.load.F_d_dir` to `Fhat`, computes `α` as `angleBetweenSymmetric(Fhat, grain)` normalised to [0, 90°] (so the engine is symmetric under load-direction reversal — the spacing rules don't care which way along grain the load points), records `F·grain` and `F·perp` for the inspector; (3) classify each of the four AABB edges via `classifyEdgeByLoad(Fhat, edgeNormal)` from `02e-catalogue-lookups.js` — strict dot-product test, boundary case (F·n exactly zero) classes as 'unloaded' to match Dan's hand calc and the Rothoblaas p. 215 figure ranges; (4) resolve required distances by linearly interpolating each of a₁/a₂/a₃,t/a₃,c/a₄,t/a₄,c between the rule set's α=0° and α=90° endpoint formulas via `interpRuleAtAlpha`, then applying Douglas-fir × 1.5 on a₁/a₃ and timber-to-timber × 1.5 on a₁/a₂ in the documented order; (5) per-screw edge-distance checks — for each screw × each edge, project the screw onto the AABB face along the outward normal to get the applied distance, look up the required value based on the edge's structural classification (end → a3t/a3c; side → a4t/a4c), emit a structured row with `applied`, `required`, `formula`, `pass`, `severity`; (6) pair-wise spacing — for every (i, j) pair project the inter-screw vector into the grain frame to get Δ_along and Δ_cross, check the EN 1995-1-1 / Rothoblaas rule that a pair PASSES if either |Δ_along| ≥ a₁_req OR |Δ_cross| ≥ a₂_req (not both — only fails if inside the (a₁, a₂) clearance rectangle in the grain frame), records the governing condition for the inspector; (7) n_ef row detection by clustering screws on their grain-projected `cross` coordinate with ε = a₂_req/2 (so two screws in the same row along grain land in the same cluster, two in different rows do not), sorting each cluster along `along`, finding the minimum consecutive spacing as a₁, looking up `lerpNef(n, a₁/d)` from the Rothoblaas table for n ∈ [2,5] with the EN 1995-1-1 §8.3.1.1 formula extension for n > 5; (8) capacity chain — looks up R_V,k from `eps90_alpha90` (= p. 216 — the conservative ε/α placeholder per Q1 in `08-open-questions.md`, with a `warnings` entry flagging this), multiplies by k_dens, by k_mod, divides by γM to get per-screw R_V,d, then for each detected row multiplies by n_ef and sums across rows to get R_V,d,total, computes η = F_d / R_V,d,total, populates `multiplierTrace` with every intermediate value for the inspector breakdown;
