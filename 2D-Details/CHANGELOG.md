### Added

- **Timber‑screw connection feature — Phase 4: rule engine (`dev/` only) — v1 acceptance gate MET**. New file `79-checks-timber.js` (~440 lines) implementing the pure-function `checkConnection(connection, model?) → ConnectionCheckResult`. Algorithm walks the nine steps from `dev/feature-timber-screws/03-rule-engine.md` exactly: (1) resolve rule context — picks the right `RuleSet` and `CapacityTable` from the catalogue based on `connectionType` × substrate family × `preDrilled`, looks up `k_mod` from EN 1995-1-1 Table 3.1 via service class × load duration, looks up `k_dens` from the rear member's `TimberClass`, fails fast with a clear `failingItems` entry if any lookup misses; (2) build geometric reference frame — normalises `rear.grainDir` to unit `grain`, derives `perp` as 90° CCW, normalises `connection.load.F_d_dir` to `Fhat`, computes `α` as `angleBetweenSymmetric(Fhat, grain)` normalised to [0, 90°] (so the engine is symmetric under load-direction reversal — the spacing rules don't care which way along grain the load points), records `F·grain` and `F·perp` for the inspector; (3) classify each of the four AABB edges via `classifyEdgeByLoad(Fhat, edgeNormal)` from `02e-catalogue-lookups.js` — strict dot-product test, boundary case (F·n exactly zero) classes as 'unloaded' to match Dan's hand calc and the Rothoblaas p. 215 figure ranges; (4) resolve required distances by linearly interpolating each of a₁/a₂/a₃,t/a₃,c/a₄,t/a₄,c between the rule set's α=0° and α=90° endpoint formulas via `interpRuleAtAlpha`, then applying Douglas-fir × 1.5 on a₁/a₃ and timber-to-timber × 1.5 on a₁/a₂ in the documented order; (5) per-screw edge-distance checks — for each screw × each edge, project the screw onto the AABB face along the outward normal to get the applied distance, look up the required value based on the edge's structural classification (end → a3t/a3c; side → a4t/a4c), emit a structured row with `applied`, `required`, `formula`, `pass`, `severity`; (6) pair-wise spacing — for every (i, j) pair project the inter-screw vector into the grain frame to get Δ_along and Δ_cross, check the EN 1995-1-1 / Rothoblaas rule that a pair PASSES if either |Δ_along| ≥ a₁_req OR |Δ_cross| ≥ a₂_req (not both — only fails if inside the (a₁, a₂) clearance rectangle in the grain frame), records the governing condition for the inspector; (7) n_ef row detection by clustering screws on their grain-projected `cross` coordinate with ε = a₂_req/2 (so two screws in the same row along grain land in the same cluster, two in different rows do not), sorting each cluster along `along`, finding the minimum consecutive spacing as a₁, looking up `lerpNef(n, a₁/d)` from the Rothoblaas table for n ∈ [2,5] with the EN 1995-1-1 §8.3.1.1 formula extension for n > 5; (8) capacity chain — looks up R_V,k from `eps90_alpha90` (= p. 216 — the conservative ε/α placeholder per Q1 in `08-open-questions.md`, with a `warnings` entry flagging this), multiplies by k_dens, by k_mod, divides by γM to get per-screw R_V,d, then for each detected row multiplies by n_ef and sums across rows to get R_V,d,total, computes η = F_d / R_V,d,total, populates `multiplierTrace` with every intermediate value for the inspector breakdown; (9) roll up — emits `overall: 'PASS' | 'FAIL' | 'WARN' | 'ERROR'` based on any edge fail OR any pair fail OR capacity overload OR (else) presence of warnings, with `failingItems[]` carrying human-readable detail strings for every failing rule. Mixed-diameter guard runs upstream: any connection with screws of differing `d` short-circuits to `overall: 'ERROR'` with a clear failingItems entry (v1 scope per `08-open-questions.md`). Console helper `tmbrCheckExampleConnection()` finds the most-recently-created Connection in the elevation bucket, runs the engine, prints a one-screen digest (overall + η + α + multiplier trace + detected rows + any failing items + any warnings) and caches the result on `connection.checks` for the Phase-6 inspector to read. **v1 acceptance gate met — 74/74 assertions across 8 test cases**: Test 1 (Dan's worked example) reproduces every expected number — α=0.000°, ε=90.000°, 24/24 edge checks pass with exact `applied` and `required` values per the hand calc (s1 bottom: applied=200 vs req=144 mm = 12·d; s1 top: applied=1020 vs req=84 mm = 7·d; s1 left/right: applied=120/220 vs req=36 mm = 3·d), 15/15 pair checks pass, both rows detect with n=3, a₁=60 mm = 5·d, n_ef=1.860 (matching the Rothoblaas table exactly), capacity chain reproduces R_V,k=12.99 → R_V,d per screw=8.394 → R_V,d total=31.224 kN → η=0.801; Test 2 (α=90°) confirms edge reclassification — right edge becomes stressed (a4t = 7d = 84 mm), left unloaded (a4c = 3d = 36 mm), top/bottom unloaded ends (a3c = 7d = 84 mm); Test 3 (no pre-drilling) correctly fails 4 within-row pairs because a₁ at α=0° no-predrill = 12·d·0.7 = 100.8 mm and the 100 mm in-row spacing falls 0.8 mm short; Test 7 (F=100 kN) correctly fails capacity at η=3.20 with a `capacity` failing-items entry; Test 9 (single screw) returns 0 pair checks, 1 row with n=1 and n_ef=1, R_V,d total=8.394 kN, η=2.98 FAIL; Test 11 (mixed d=12 and d=10) returns `overall: 'ERROR'` with the mixed-diameter failingItems entry; Test 12 (SC3 medium) drops k_mod to 0.65 → R_V,d=6.82 per screw → total=25.37 kN → η=0.985 just passes; Test 13 (SC1 permanent) drops k_mod to 0.60 → η=1.068 FAIL. Engine returns `overall: 'WARN'` rather than `'PASS'` for the gate case because of the documented ε/α placeholder warning — that will return to `'PASS'` automatically when Q1 in `08-open-questions.md` resolves and the warning is removed. `dev/index.html` updated with the new script tag. No edits to existing files needed (engine is pure — call sites come in Phase 5 with the canvas-overlay renderer).

- **Timber‑screw connection feature — Phase 3: Connection entity (`dev/` only)**. New file `78-connection.js` (~290 lines) defining the `connection` entity type — the structural grouping of a front entity (steel plate or timber member), a rear timber member, a list of screw IDs, and the applied load vector. Factory `mkConnection(viewKey, frontEnt, rearEnt, opts)` defaults to steel-to-timber + pre-drilled + 25 kN downward + medium-term + SC1 (these are the v1 sensible defaults set in Phase 0). `inferConnectionType()` resolves the connection type from the front/rear entity types so the caller doesn't need to specify (steel-plate + timber-member → steel-to-timber; timber-member + timber-member → timber-to-timber stub for v1.1; safe steel-to-timber fallback otherwise). Binding API: `tmbrBindScrew(conn, screwEnt)` sets `screw.connectionId` and appends to `conn.screws[]` idempotently with auto-unbind from any previous connection; `tmbrUnbindScrewFromCurrent(screwEnt)` is the inverse; `tmbrSetLoad(conn, F_d_dir, F_d_mag, opts)` normalises the direction to a unit vector, optionally updates loadCase/serviceClass, and invalidates `conn.checks` so the Phase-4 engine re-runs; `tmbrSetPreDrilled(conn, value)` toggles + invalidates. Lookups: `tmbrGetConnection(viewKey, id)`, `tmbrGetEntityById(viewKey, id)`, `tmbrGetScrewsForConnection(conn)`, `tmbrConnectionAnchor(conn)` returns the load-arrow origin at the rear member's centroid. Teardown: `tmbrDeleteConnection(conn)` removes the entity from `entities2D[viewKey]` and clears `connectionId` on every bound screw. Renderer `drawConnection(blk, ent, cs)` paints two layers: (1) a dashed `--accent`-coloured inset highlight on each bound member's outline (4 mm inset) so the engineer can see at a glance which entities belong to which connection; (2) the load arrow itself — a thick shaft pointing along `F_d_dir` from the rear centroid, an equilateral filled triangle arrowhead (20 mm head length × 12 mm head width), and a perpendicular-offset bold label with the kN magnitude plus a muted sub-label showing loadCase • serviceClass. Console helper `tmbrCreateExampleConnection()` builds Dan's full Test 1 fixture in one call — column + plate + connection (with default 25 kN downward load) + 6 screws bound — and assigns `tmbrCurrentConnectionId` so a follow-on screw-tool will bind to it. `'connection'` dispatch case added to `34-draw-2d.js` immediately after the Phase 2 cases, gated on the same `FEATURE_TIMBER_SCREWS` flag. `dev/index.html` updated. 36 assertions pass in the headless verification (refs, binding state, load normalisation including non-unit input normalisation to magnitude 1, anchor at column centroid (170, 610) = (w/2, h/2), rule-set version stamp `rothoblaas-hbs-plate-eta-11-0030-2019`, JSON serialise/parse round-trip preserves every field including the front/rear/screws refs and the load object, setLoad invalidates `conn.checks`, setPreDrilled invalidates, unbind shrinks `screws[]` and clears `screw.connectionId`, rebind restores both, deleteConnection removes the entity AND clears connectionId on all formerly-bound screws). SVG at `dev/feature-timber-screws/verification/phase3-fixture.svg` (87 KB, 635 SVG elements) shows the load arrow + member-highlight overlaying the full Dan Test 1 geometry. **Click-tool integration deferred to Phase 3.5** (timber-member / steel-plate / screw placement tools + start-connection action + load-arrow tool) — bundles cleanly with the eventual `39-events.js` dispatch-tree refactor called out in known-issue #2 in the project root CLAUDE.md; Phase 3 ships the programmatic API which is sufficient for testing and lets Phase 4's rule engine proceed in parallel.

### Changed

- V25 cross-section placement is now single-click (was: two-click drag, same as elevation). Picking "Cross-section" in the Aspect dropdown switches the v25-mem tool to a click-to-place flow: a single canvas click drops the section at the cursor with `length: 0`, `rot: 0`, `aspect: 'sec'`. Default `rot: 0` gives the natural drafting orientation — web vertical, flanges horizontal — instead of the meaningless `atan2(dy, dx)` value the previous drag flow stamped from the user's drag direction. PFC sections carry the current `v25State.openSide`. New entity auto-selects so the inspector opens immediately. Elevation aspect is unchanged (still two-click length + angle). Implementation in the `v25-mem` branch of `v25TryHandleClick` (`69-v25-dispatch.js`).
- V25 cross-section placement now shows a ghost preview of the actual section under the cursor while in v25-mem with Aspect = "Cross-section". The preview is a transient `mem2` entity passed through the real `drawMem2D` pipeline at `opacity: 0.45`, so it's pixel-identical to what placement produces (no duplicate geometry to keep in sync). Implementation extends `v25DrawPreview` in `69-v25-dispatch.js`. Aspect dropdown change in `72-v25-options-bar.js` now also triggers a `requestRender()` so the ghost appears the moment the user switches to cross-section, without needing to nudge the mouse first.

### Fixed

- V25 cross-section mem2 drag no longer stretches the section into a length-having elevation member. Cross-sections (`aspect: 'sec'`, `length: 0`) had `end-a` and `end-b` handles both emitted at the entity origin by `v25EntHandles`, so any body-drag click landed on `end-a` within tolerance, ran the end-handle branch in `v25Move`, and re-computed `ent.length = hypot(du, dv)` and `ent.rot = atan2(dv, du)` on every mousemove — looking to the user like the section was rotating wildly. `v25EntHandles` and `v25HitHandle` in `71-v25-selection.js` now both guard on `(ent.aspect || 'elev') === 'sec' || length < 1` and skip end-handle emission / classification for cross-sections. Body translation now works as expected (uses the existing `v25Move` body branch which just translates `ent.u`/`ent.v`). Rotating a cross-section still goes through the inspector — on-canvas rotate handle for `aspect: 'sec'` is a separate enhancement.

### Added

- V25 plate2 grip-drag Shift modifier — symmetric inverse semantics so the same key swaps the default behaviour for both shapes.
  - **Rectangle plate**: default corner drag = rectangle resize with opposite corner anchored (`v25Move` recomputes `ent.u/v/w/h` from min/max of the four corners). **Hold Shift** = morph the entity in place to `shape: 'poly'` with the four corners frozen as `pts[BL, BR, TR, TL]`, then translate only the dragged corner — distorts the rectangle into a freeform quadrilateral. `v25Drag.handle` is re-keyed from `corner:XX` to `pt:N` mid-drag so subsequent mousemoves continue through the polygon pipeline.
  - **Polygon plate**: default vertex drag = translate only the picked vertex (same as before). **Hold Shift** = bbox-scale all vertices proportionally around the OPPOSITE bbox corner (the dragged vertex's position dictates the new scale; degenerate vertices on the anchor side fall back to no scaling on that axis). Symmetric inverse of rect+Shift.
- V25 plate2 body-drag soft-snap to mem2 outer faces — when translating a placed plate via body grip, the plate's axis-aligned edges soft-snap to nearby mem2 outer faces (and to other axis-aligned plate edges). 2 mm catch zone, 3 mm release zone, matching the existing mem2↔mem2 snap. Polymorphic extension of `v25ApplySnap` in `68-v25-tools.js` (new `edgesFor(ent)` dispatcher handles both types; collector loop now walks mem2 + plate2). New `v25Plate2EdgesForSnap(ent)` in `76-v25-plate.js` emits `{axis, value, label}` bbox edges only when the plate orientation is within ±5° of an axis. `applyU/applyV` translation helpers now also translate `ent.pts[]` so polygon plates snap correctly without leaving their outline behind. Body-snap gate in `39-events.js` broadened from `type === 'mem2'` to `type === 'mem2' || type === 'plate2'`.
- V25 plate2 two-click rect placement for snapped first clicks — when the user clicks a snapped mem2 / plate edge in v25-plate Elevation mode WITHOUT dragging, instead of starting a polygon (which fires for non-snapped click-releases), the tool now enters a two-click rect flow: the snapped point becomes the first corner, a dashed rect preview follows the cursor, and the next click commits the opposite corner.
